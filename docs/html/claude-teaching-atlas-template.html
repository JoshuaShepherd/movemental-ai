How to read this atlas

Top to bottom is the conceptual stack; on wide screens the left column is your map. Product names and limits change — use this as a teaching scaffold, not procurement law.

Claude is a family of large language models from Anthropic, usually experienced through chat, API, or developer tools. In staff conversations, "Claude" often means the whole assistant experience (interface, defaults, and your inputs), not only the model weights.

For churches and nonprofits: the first risk is usually not "which model" but what entered the context, who approved it, and what habit that trains in people who turn over every season.

Skills, projects, artifacts

Three labels people conflate — keep the distinctions visible.

Skill

Packaged instructions the assistant can load for a class of tasks (for example board memo in house voice). Repeatable method, not one-off chat.

Project

A scoped workspace for a stream of work — briefs, files, continuity — still needs curation of what belongs inside.

Artifact

A side document or editable output kept alongside chat — good for drafts leaders review before they become official.

Skills lab

Why skills exist

Skills are how you turn occasional clever prompting into repeatable organizational method — with enough structure to govern, version, and retire.

Without skills, every staff member re-invents tone, steps, and guardrails in the chat box. Quality varies by fatigue. Donor-facing drafts drift. Nobody can audit what “the org’s way” actually is.

With well-designed skills, you encode the boring parts once: voice, checklist, refusal rules, output shape, and where to look for truth. The model still needs supervision — but the baseline becomes shareable.

Teaching distinction: a skill is not “a long prompt you saved.” It is a scoped capability contract: when it applies, what it must never do, what it must ask a human, and what shape the deliverable takes.

Spectrum of “skill-ness”

Same word, very different maturity. Know where you are before you invest in packaging.

Level A — Snippet libraryCopy/paste blocks in Notion or email templates. No machine-readable contract.
Ad hoc
Level B — Saved prompt / “cookbook step”One-shot instructions the human pastes when they remember. Easy; drifts fast.
Personal
Level C — Project-scoped instructionsRules that ride along with a workspace or thread: good for one initiative, still needs dedup across projects.
Scoped
Level D — Registered skill packNamed capability with discovery metadata, versioning, owners, and (ideally) test prompts + eval notes.
Org-grade
Level E — Agentic / code-coupled skillInstructions plus tools: filesystem, MCP, HTTP, DB — with logging, secrets handling, and CI. This is “platform,” not “prompting.”
Engineering

Movemental’s nonprofit/church deployments usually need to graduate teams from B → D before they touch E.

Seven layers (what stacks on what)

When two instructions disagree, the winner is usually the higher, stricter, or more recently ratified layer — but products differ; this is a teaching map, not a court ruling.

Layer 1 — Model & platform safety Classifier policies, abuse blocks, capability refusals. You do not override this with a skill.
Immutable
Layer 2 — Tenant / org policy connectors SSO claims, DLP, allowed URLs, MCP allowlists, audit sinks. Shapes what tools and retrieval are even legal.
Governance
Layer 3 — Workspace defaults Org-wide “how we write,” citation rules, brand tokens, languages, footnote style — the air everyone breathes.
Baseline
Layer 4 — Skill registry (metadata) Titles, descriptions, tags, owners, version pins. Discovery happens here; bodies load only when relevant.
Registry
Layer 5 — Activated skill body The instructions actually injected / retrieved for this run: procedures, examples, edge cases.
Active
Layer 6 — Project memory & files Uploaded PDFs, style guides, prior drafts in this initiative. Highest risk of PII creep — curate ruthlessly.
Context
Layer 7 — This-turn user message Immediate ask, corrections, “try again with…” — can contradict lower layers if the product allows; that is a feature and a foot-gun.
Ephemeral

Design rule: Put immovable law in Layer 2–3 (policy + workspace). Put method in Layer 4–5 (skill). Put facts for this case in Layer 6–7 (files + prompt).

Anti-pattern: Pasting your entire employee handbook into a skill (Layer 5) instead of summarizing handling rules into Layer 3 and linking canonical HR in a controlled repository (Layer 6).

Nonprofit note: pastoral care examples, donor stories, and youth identifiers should never live as “few-shot examples” inside a shared skill. Use synthetic or public-domain examples.

Invocation modes (options matrix)

How a skill enters a run changes cost, surprise, and auditability. Pick deliberately per workflow.

Mode When it loads Strengths Risks
Pinned / always Every message until unpinned Consistent voice; hard to “forget” the rules Context bloat; conflicting pins; stale policy
User-invoked Human selects from menu / slash / prompt Transparent; easy consent; good for rare workflows People skip it under pressure; uneven adoption
Model-routed Product may attach skill when description matches intent Low friction; feels magical for experts Wrong skill surprise; needs crisp descriptions + evals
API tool Developer supplies tool definitions + handler code Strongest guarantees; logging; typed I/O Engineering burden; security review surface
Hybrid Thin always-on “router skill” + fat specialist on demand Balances context vs coverage Router errors cascade; needs monitoring

Enterprise reality: combine user-invoked for high-stakes (finance, HR) with pinned thin baselines for voice and citation — never pin ten “fat” skills at once.

Routing, metadata, and progressive disclosure

Most “wrong skill attached” bugs are bad routing text, not bad steps. Treat the short description as an API contract.

Name (human): 2–5 words staff recognize in a menu — “Board memo coach,” not “BMv3 helper.”

Description (machine + human): One tight paragraph: triggers (inputs that must be true), deliverable, non-goals (“not legal advice”), and escalate when.

Tags / domain prefix: e.g. comms.donor-letter — makes search, ACL, and analytics honest.

Progressive disclosure tiers: (T0) registry line only. (T1) 400-token “always” preamble. (T2) full procedure loaded when invoked or routed. (T3) references/ pulled only for named edge cases.

Evaluation hooks: keep 3–5 golden prompts per skill; rerun when the base model changes; log regression deltas in CHANGELOG.

If the description could match half the org’s chat traffic, the router will — and you will get “skill spam.” Narrow wins.

Anatomy of a serious skill pack

Think like a tiny internal product: README for humans, contract for the model, tests for regressions.

Filesystem shape (Claude Code–style repos often look like this)

.claude/skills/ grant-letter-coach/ SKILL.md ← name, when-to-use, steps, guardrails references/ tone-samples.md ← optional deep docs (load on demand) scripts/ ← optional helpers (review carefully) CHANGELOG.md

SKILL.md front matter (conceptual): machine-readable description so routing does not misfire; human-readable name; optional triggers / audience tags.

Body sections we recommend: Identity (who the assistant is in this skill), Preconditions (inputs you must gather first), Procedure (ordered steps), Output schema (headings, tables, JSON), Refusal & escalate (when to stop), Quality bar (what “done” means).

Progressive disclosure: keep the always-loaded surface small; push long law and edge cases into references/ and teach the skill to pull only what applies.

Two surfaces: chat product vs developer tooling

Same word “skill,” different packaging and blast radius.

Claude (team / enterprise chat)

  • Audience: program staff, comms, leadership pilots
  • Skills as governed prompts + optional files in a workspace
  • Win: low code; fast iteration; easy demos
  • Watch: PII in uploads; pin proliferation; who can publish

Claude Code / agent repos

  • Audience: engineers, technical PMs, integrators
  • Skills as markdown + tools + symlinks to shared libraries
  • Win: composability; CI; typed boundaries
  • Watch: secrets in env; unreviewed scripts; supply chain
Teaching line for exec briefings: Chat skills shape how people work. Code skills shape how software is built. Nonprofits usually need both — but different approvers and different SLAs.

Precedence, conflicts, namespaces

When two skills both “fit,” ambiguity becomes a random number generator.

Namespace discipline: prefix skills by domain — fin., hr., comms. — so humans and routers parse intent faster.

Precedence habits: (1) Global workspace rules beat generic skills. (2) Narrow skills beat broad “god skills.” (3) Newer published version beats draft — never let two semver ranges float silently.

Conflict resolution meeting: once per quarter, print the top 10 skills by usage; search for overlapping descriptions; merge or split.

Lifecycle: cradle to archive

Skills rot faster than blog posts. Treat them like internal software.

0 — SparkSomeone ships a great one-off prompt in a crisis. Capture it raw (no PII).
1 — Draft skillPeer review with checklist (below). Pilot with 2–3 trusted users.
2 — Publish v1Semver tag, owner, support channel, deprecation date if experimental.
3 — InstrumentLight telemetry: optional thumbs + “what failed?” taxonomy — not surveillance.
4 — Revise / splitIf failure clusters around one step, fork a specialized child skill instead of bloating parent.
5 — ArchiveRetire with redirect note to replacement; never silent-delete.

Governance & data classes per skill

Attach a data label to every skill the way you label databases.

Public-safe

May be demonstrated in all-staff training. Examples use synthetic donors.

Internal-only

May reference real policy titles but not case files. No volunteer access by default.

Restricted

Finance, HR, legal: dual control on edits; quarterly access review.

Ephemeral assist

Never stores examples from live runs; instruct user to paste redacted snippets only.

Composition patterns

How skills meet retrieval, humans, and other skills without spaghetti.

Skill + approved corpus

  • Skill states how to reason; RAG supplies facts.
  • Never let the skill “invent” policy — quote corpus or escalate.

Thin router → fat specialist

  • Router classifies intent; loads one specialist skill body.
  • Prevents loading five 30-page skills “just in case.”

Human-in-the-loop gates

  • Skill ends at draft + checklist; human clicks “approved to send.”
  • Best pattern for donor-facing comms in regulated orgs.

Handoff to specialist staff

  • Skill outputs a structured handoff packet (facts, unknowns, risks).
  • Counselor / attorney / ED takes over — model stops.

Failure modes (recognize early)

Context obesity: skills so long that your actual question gets starved — latency up, quality down.

Policy archaeology: copying 2022 board PDFs into a skill instead of summarizing live policy into three non-negotiables.

Skill sprawl: 40 overlapping “writing helpers” — pick winners or staff will roll dice.

Example poisoning: real beneficiary quotes embedded “for quality” — irreversible overshare risk.

Silent routing errors: vague descriptions → wrong skill attached → confident wrong answer.
