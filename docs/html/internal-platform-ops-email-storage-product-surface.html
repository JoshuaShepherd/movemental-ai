2. Supabase Storage and public/images — best practice in this stack

Two layers (both valid)

  • public/images/… — Static assets shipped with the Next.js app (SVG logos, small icons, anything that rarely changes and does not need a CDN outside the deployment). Good for source files before upload (e.g. optimized WebP in public/images/voices/).
  • Supabase Storage (public bucket) — CDN-backed URLs for larger or frequently updated media, multi-tenant paths, and assets shared across apps. This repo uses bucket media-library with paths like movemental/voices/*.webp and full URLs of the form https://<project>.supabase.co/storage/v1/object/public/media-library/….

Upload pipeline (pattern in this repo)

Script: scripts/upload-voices-to-supabase-storage.ts — reads from public/images/voices/*.webp, uploads with service role, upsert: true, correct contentType. Run locally with .env.local containing NEXT_PUBLIC_SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY.

Next.js + Tailwind consumption

  • Allow remote images in next.config.ts under images.remotePatterns for *.supabase.co with pathname /storage/** (already present).
  • Use next/image with explicit width/height or fill and a correct sizes attribute for responsive layouts (see components such as src/components/sections/voices/voice-portrait.tsx).
  • Prefer WebP (or AVIF where supported) at sensible dimensions; avoid shipping multi-megabyte originals to the browser.
  • Keep tenant-scoped folders in the bucket (movemental/…, alan-hirsch/…) so RLS and human navigation stay clear.
  • For private assets, use a non-public bucket and signed URLs — do not expose sensitive files on …/object/public/….

Tailwind / design tokens

Images are still wrapped in semantic layout (e.g. rounded cards, aspect-* boxes). Per Movemental design rules, avoid arbitrary hex on image frames; use token classes (bg-card, border-border, etc.) in TSX — the HTML file cannot import Tailwind, but components should.

3. Walkthrough vs demo vs views — when and how (UI + design)

For a serious B2B / institution-facing platform (Alan Hirsch–style tenant), buyers need three different proofs. They answer different anxieties; mixing them up creates “demo theater” without trust.

Definitions

  • Walkthrough — Narrated, often linear explanation of how work flows through the product (problem → action → outcome). Can be live (Zoom + screen share), recorded (Loom / MP4), or a guided in-product tour (step highlights). Best for judgment and fit: “Do we actually work this way?”
  • Demo — Controlled scenario showing capabilities in ideal conditions (sample data, happy path). Best for breadth and wow: “What does this thing do?” Higher risk of feeling synthetic if it is the only surface.
  • Views (live build / “see it in the wild”) — The real deployed product (or read-only shadow) that prospects can click themselves or that you point to as evidence. Best for authenticity and craft: “This is not a mock.”

When to use each

Buyer stageLead withSupport with
Cold / curious Short demo clip or hero loop (under 2 minutes) Link to live view for one or two safe pages (marketing, public library)
Evaluation / committee Structured walkthrough mapped to their jobs-to-be-done Demo data only where it clarifies; otherwise live view
Security / procurement Architecture and data-flow narrative (often doc-led) Screenshots or short walkthrough clips with no PII

UI and design guidance (concrete)

  • Hero / technology page: Use a single aspect-video container with clear label: e.g. “Hero walkthrough loop” + subcopy that says it is a cinematic MP4 (dashboard, workflow, AI lab, CMS). Primary CTA: Book a live walkthrough; secondary: See a live build (external link). This mirrors the pattern on src/components/studio/pages/TechnologyPage.tsx.
  • Capability rows: Pair each block with a thumbnail or loop placeholder plus text; use “Watch the walkthrough” as a text button with play affordance, not only a mystery click target.
  • Live builds grid: Card with subtle “Live” indicator (pulse dot), real URL, and explicit line: live platform, not a sandbox demo — reduces cynicism.
  • Contact / positioning pages: When the brand promise is conversation-first (no slide deck), say so plainly; do not bury the user in autoplay video.
  • Motion: Respect prefers-reduced-motion; do not autoplay heavy video above the fold without mute and a pause control.
Rule of thumb: Demo earns attention; walkthrough earns understanding; live view earns trust. Ship all three in different UI zones rather than collapsing them into one generic “video.”
